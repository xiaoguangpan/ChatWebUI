# ChatWebUI 产品界面规范

ChatWebUI 的界面由用户端和运营后台组成。用户端强调连续对话、历史恢复和模型选择效率；运营后台强调表格浏览、筛选、批量操作和状态可见性。整体视觉使用中性背景、清晰层级和少量品牌色，支持浅色和深色主题。

## 设计目标

- 对话页面保持低干扰，输入、模型选择、历史切换路径明确。
- 生图页面保留生成状态、结果预览、重试和删除能力。
- 后台页面适合长时间运营使用，信息密度高但层级清晰。
- 表格区域优先使用内部滚动，避免分页和刷新按钮被长内容推离视口。
- 主题切换后按钮、弹框、Markdown、Mermaid 图表都保持可读。

## 主题与颜色

系统通过 CSS 变量控制主题，入口文件为 `apps/web/src/styles/tokens.css`。

核心背景：

| Token | 深色 | 浅色 | 用途 |
| --- | --- | --- | --- |
| `--bg-canvas` | `#212121` | `#FFFFFF` | 主画布 |
| `--bg-sidebar` | `#181818` | `#F4F4F4` | 侧边栏 |
| `--bg-elevated` | `#2F2F2F` | `#FFFFFF` | 弹框、卡片、输入面 |
| `--bg-hover` | `#2A2A2A` | `#ECECEC` | Hover 状态 |
| `--bg-active` | `#3A3A3A` | `#E4E4E4` | 选中状态 |

核心文本：

| Token | 深色 | 浅色 | 用途 |
| --- | --- | --- | --- |
| `--text-primary` | `#ECECEC` | `#0D0D0D` | 主要文本 |
| `--text-secondary` | `#B4B4B4` | `#5D5D5D` | 次要文本 |
| `--text-tertiary` | `#8E8E8E` | `#8E8E8E` | 占位和辅助信息 |
| `--text-link` | `#7FB7FF` | `#1A56DB` | 链接 |

功能色：

| Token | 用途 |
| --- | --- |
| `--brand` | 主按钮、强调操作 |
| `--success` | 成功、可用、正常 |
| `--warning` | 警告、等待、波动 |
| `--danger` | 错误、失败、删除 |
| `--info` | 信息提示 |

## 排版

- 字体：系统 UI 字体优先，中文兼容 `PingFang SC`、`Microsoft YaHei`。
- 正文：14px，行高 1.6。
- 对话正文：16px，行高 1.6。
- 紧凑后台表格：12px 到 14px。
- 字间距保持 0，避免中文排版失真。

常用字号：

| Token | 尺寸 | 用途 |
| --- | --- | --- |
| `--fs-h1` | 28px | 页面主标题 |
| `--fs-h2` | 24px | 模块标题 |
| `--fs-h3` | 20px | 面板标题 |
| `--fs-body-lg` | 16px | 对话内容 |
| `--fs-body` | 14px | 表单、表格、正文 |
| `--fs-caption` | 12px | 标签、时间、辅助信息 |

## 布局

用户端：

- 桌面端左侧固定侧栏，主区域显示聊天或生图内容。
- 移动端侧栏使用抽屉，主操作保持在底部输入区域附近。
- 聊天内容最大宽度保持在适合阅读的范围内。
- 新会话和历史会话切换必须保留清晰上下文。

运营后台：

- 左侧固定导航，右侧为页面内容。
- 仪表盘允许整页纵向滚动。
- 表格类页面优先让表格容器内部滚动。
- 筛选栏、批量操作、分页和刷新按钮应保持稳定位置。
- 长错误信息必须截断或进入详情，不允许挤压其他列。

## 组件规范

按钮：

- 主操作使用 `c-btn--primary`。
- 次操作使用 `c-btn--secondary`。
- 危险操作使用 `c-btn--danger`。
- 图标按钮必须有 `aria-label`。
- 浅色模式下主按钮必须保持品牌底色和白色文字。

输入：

- 普通输入高度 44px。
- 大输入高度 52px。
- 输入错误状态使用红色边框和简短错误文本。
- 聊天输入框支持多行，超出最大高度后内部滚动。

弹框：

- 表单弹框底部保留确认和取消按钮。
- 危险操作需要明确确认。
- 弹框关闭后不保留隐藏的大量 DOM，避免页面卡顿。

表格：

- 操作列靠右。
- 用户 ID、错误信息、UA、路径等长文本必须截断。
- 可导出页面使用统一 CSV 导出工具。
- 空状态也要保持表格区域高度，避免按钮位置跳动。

Badge：

- 成功、失败、测试中、未测试、可见性、默认模型使用不同语义色。
- 默认模型应在模型列表中明显标识。

## 对话内容渲染

- Markdown 使用统一渲染组件。
- 代码块、表格、链接和列表保持可读。
- 外部链接默认新窗口打开。
- Mermaid 图表按照当前主题重新初始化。
- 流式输出中避免每个字符触发重型图表渲染导致闪烁。

## 模型服务页面

- 供应商接入、全部模型、积分策略为同一模块的三个主视图。
- 添加模型时远端模型列表必须支持滚动和筛选。
- 已导入模型显示为已勾选；取消勾选表示删除导入模型。
- 批量测试默认排除图片模型，图片模型单独测试时需要提示耗时和额度风险。
- 全部模型按文字、图片、语音、其他分类显示。
- 默认模型只能从所有人可见模型中选择。

## 生图页面

- 发送指令后立即显示生成中状态。
- 生成成功后进入生图历史。
- 生成失败保留失败记录和错误信息。
- 图片支持点击放大查看。
- 支持重新生成和删除。

## 后台用户详情

- 用户详情包含资料、会话、生成记录、积分记录、登录历史。
- 会话记录应展示完整对话内容，渲染方式与用户端一致。
- 管理员重置密码、调整积分、禁用用户必须有清晰反馈。

## 响应式要求

- 关键按钮文字不能溢出容器。
- 小屏下表格可横向滚动，不能压缩到不可读。
- 固定格式控件使用稳定尺寸，避免 hover 或加载状态造成布局跳动。
- PC 个人中心二级页面必须提供返回入口。

## 可访问性

- 所有按钮和菜单可键盘聚焦。
- 图标按钮提供 `aria-label`。
- 表单错误信息和输入控件有关联。
- 颜色不能作为唯一状态提示。
- 动效遵循 `prefers-reduced-motion`。
